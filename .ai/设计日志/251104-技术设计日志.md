## 251104 讨论：简化 Monorepo 架构，转向独立的 Variant 开发模式

### 背景

在实践了 [251017 讨论的工程化方案](./技术设计笔记-251010.md#251017-讨论工程化与多版本并行开发方案) 后，我们发现该方案对于项目当前的早期探索阶段而言过于复杂。`shared` 和 `tooling` 等包的划分引入了不必要的管理开销，并且在 `variant` 的 API 和需求快速变化的情况下，并没有出现真正稳定可复用的部分，导致迭代速度受阻。

### 决策

为了重新聚焦于核心功能的快速原型开发，我们决定**大幅简化 Monorepo 架构**。

核心变更如下：
1.  **移除共享包**：废除 `shared/` 和 `tooling/` 目录，不再强制进行跨 `variant` 的代码共享。
2.  **Variant 独立化**：每个 `variant-drafts` 下的包都将成为一个完全独立的、自包含的项目。如果需要，它可以拥有自己的工具链（如 Vite 插件）。
3.  **代码复用降级**：在探索阶段，代码复用将通过临时的手动复制代码来实现。当未来出现明确、稳定的共享需求时，再重新提取共享包。

这个决策旨在降低项目的工程复杂度，遵循 YAGNI (You Ain't Gonna Need It) 原则，避免过早抽象。
